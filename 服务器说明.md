# 服务器说明

这份文档说明：把当前项目上传到服务器之后，建议按什么顺序准备环境、重建数据清单、训练和验证。

## 1. 建议先做的事

进入项目目录：

```bash
cd /your/path/UNET_two_stage
```

创建虚拟环境并安装依赖：

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

检查 PyTorch 是否识别 GPU：

```bash
python -c "import torch; print(torch.cuda.is_available(), torch.cuda.device_count())"
```

如果输出是 `True` 且设备数大于 0，说明 CUDA 可以用。  
如果服务器没有 GPU，而配置文件里仍然是 `device: cuda`，训练会直接报错。

## 2. wandb

如果你想记录训练过程到 Weights & Biases：

```bash
wandb login
```

当前项目默认配置里：

- `wandb_enable: true`
- `wandb_mode: online`

如果服务器不能联网，或者你暂时不想同步到云端，可以改成离线模式：

```bash
export WANDB_MODE=offline
```

或者直接把配置里的 `wandb_mode` 改成 `offline`。

## 3. 上服务器后是否要重新生成 manifests

建议：**要重新生成一遍。**

原因：

- `manifests/*.csv` 依赖当前环境下的数据文件和映射表
- 只要服务器上的数据路径、数据数量、映射文件、或数据版本和本地略有不同，`manifests` 就可能失效
- 当前项目里 `prepare_dataset.py` 还负责生成独立的 normal folds

因此，上服务器后建议至少重新执行：

```bash
python scripts/prepare_dataset.py
python scripts/build_patch_index.py --config configs/stage1.yaml
```

## 4. 推荐运行顺序

### 4.1 先检查数据准备

```bash
python scripts/prepare_dataset.py
python scripts/build_patch_index.py --config configs/stage1.yaml
```

这两步会重建：

- `manifests/defect_fold*.csv`
- `manifests/normal_fold*.csv`
- `manifests/normal_future_holdout.csv`
- `manifests/stage1_fold*_train_index.csv`
- `manifests/stage1_fold*_val_index.csv`

## 5. 先只跑一个 fold

不要一开始就直接跑完整 4 折。  
建议先跑 `fold 0`，确认环境、路径、CUDA、wandb 和数据准备都正常。

```bash
python scripts/train_stage1.py --config configs/stage1.yaml --fold 0
python scripts/train_stage2.py --config configs/stage2.yaml --fold 0
python scripts/evaluate_val.py --config configs/stage2.yaml --fold 0
python scripts/infer_holdout.py --config configs/stage2.yaml --fold 0
```

这一轮主要检查：

- 数据路径是否正确
- Stage1 patch index 是否正常
- Stage1 是否能正常保存 checkpoint
- Stage1 replay 是否能在训练中刷新
- Stage2 是否能加载 Stage1 最优权重
- Stage2 hard-normal pool 是否能正常生成
- 验证和 holdout 推理是否能正常输出

## 6. 确认没问题后再跑完整 4 折

### 6.1 Stage1

```bash
for fold in 0 1 2 3; do
  python scripts/train_stage1.py --config configs/stage1.yaml --fold "$fold"
done
```

### 6.2 Stage2

```bash
for fold in 0 1 2 3; do
```

### 6.3 验证

```bash
for fold in 0 1 2 3; do
```

### 6.4 holdout 推理

```bash
for fold in 0 1 2 3; do
```

## 8. 训练输出在什么位置

Stage1 输出目录：

```text
outputs/stage1/fold{fold}/
```

其中常见文件包括：

- `best_stage1.pt`
- `last_stage1.pt`
- `history.csv`
- `replay/replay_latest.csv`
- `replay/replay_latest_summary.json`

Stage2 输出目录：

```text
outputs/stage2/fold{fold}/
```

其中常见文件包括：

- `best_stage2.pt`
- `last_stage2.pt`
- `history.csv`
- `val_metrics.json`
- `val_per_image.csv`
- `val_postprocess_search.csv`
- `hard_normal/hard_normal_latest.csv`
- `hard_normal/hard_normal_latest_summary.json`
- `holdout/prob_maps/`
- `holdout/masks/`

## 9. 哪些情况下要重新跑数据准备

出现下面任一情况，建议重新执行：

```bash
python scripts/prepare_dataset.py
python scripts/build_patch_index.py --config configs/stage1.yaml
```

适用场景：

- 服务器上的数据和本地不完全一致
- 数据文件有新增或删除
- 映射表有变化
- 你改了 fold 逻辑
- 你改了 Stage1 patch 配置
- 你怀疑当前 `manifests` 与代码版本不一致

## 10. 推荐最短执行路径

如果你只想先验证服务器链路通不通，建议直接按这个顺序：

```bash
python scripts/prepare_dataset.py
python scripts/build_patch_index.py --config configs/stage1.yaml
python scripts/train_stage1.py --config configs/stage1.yaml --fold 0
python scripts/train_stage2.py --config configs/stage2.yaml --fold 0
python scripts/evaluate_val.py --config configs/stage2.yaml --fold 0
```

确认 `fold 0` 没问题后，再开始跑完整 4 折。
